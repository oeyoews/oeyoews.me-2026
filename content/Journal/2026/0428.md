## 我开始把 Agent 当同事

今天最大的变化，不是我多写了多少代码，而是我开始认真把 Agent 当成协作对象，而不是工具栏里的“增强按钮”。

以前我的工作流是：自己先写，再让 AI 修。
现在逐渐变成：先给 Agent 明确上下文和目标，再由它推进实现，我负责校验方向和质量。

这个转变很关键。它让我意识到，真正稀缺的不是“敲代码速度”，而是“定义任务边界”的能力。

## Agent 真正改变了什么

我现在对 Agent 的理解更接近三层：

1. **执行层**：快速生成样板、补全重复逻辑、整理基础结构。
2. **协作层**：在有上下文的前提下，能持续推进一个多步骤任务。
3. **决策层（仍需人主导）**：需求取舍、架构边界、长期维护成本，依然必须由人拍板。

所以，Agent 不是替代工程师，而是把工程师从低价值重复劳动里抽离出来，逼着我们把精力放到更高阶的判断上。

## 我踩过的坑

这几天最明显的坑有三个：

- 目标太宽泛：一句“帮我优化一下”会得到看起来正确、但不可验收的结果。
- 上下文不完整：不给限制条件，Agent 会默认“最佳实践”，但未必适合当前项目。
- 验收标准缺失：只看“能跑”不看“可维护”，后续返工成本更高。

这些坑本质上都指向同一件事：**Agent 的上限，取决于人的问题表达和质量门槛**。

## 我给自己定的协作原则

为了让 Agent 真正稳定产出，我准备固定这套最小协作模板：

- 先写清楚目标：要解决什么问题，不做什么。
- 再给关键约束：技术栈、风格、边界、不可变条件。
- 规定交付格式：改哪些文件、为什么改、怎么验证。
- 最后做结果验收：功能正确、结构清晰、可维护、可回滚。

一句话总结：
**会用 Agent，不是会下命令，而是会做工程管理。**
